## What this project is

**Beacon Lakehouse** is a production-grade, locally-runnable unified analytical
lakehouse built for a local-business marketing analytics domain. It demonstrates
the full data engineering stack expected of a Senior AI Data Engineer:
multi-source ingestion, medallion architecture, dbt transforms with data
contracts, orchestration with visible SLAs, data quality gating, and a
low-latency context API designed to serve AI agents.

The project is intentionally **public and reusable**. Anyone must be able to
clone it and run the full pipeline end-to-end with `make up && make seed &&
make pipeline` вЂ” zero cloud accounts, zero paid services, zero manual setup.

**Resume signal:** Every design decision maps directly to the language in
senior AI data engineering job descriptions: unified analytical layer, trusted
source of truth, data contracts, SLAs for freshness and availability, data
access patterns for AI agents, lakehouse architecture, partitioning,
governance, dbt, Airflow/Dagster, Spark/Delta, Snowflake/DuckDB.

## Domain

**Local-business marketing analytics.** The fictional company is a marketing
platform serving local businesses (think: law firms, dental practices, home
services). Data covers:

- **Clients** вЂ” businesses on the platform (onboard date, plan tier, industry,
  location)
- **Leads** вЂ” inbound inquiries generated for each client (source, status,
  value, timestamp)
- **SEO rankings** вЂ” weekly keyword position snapshots per client
- **Ad spend** вЂ” daily spend + impressions + clicks per client per channel
- **Appointments** вЂ” scheduled appointments converted from leads

This domain was chosen because it produces realistic, relatable data across
multiple source systems (operational DB, REST API, flat files) and maps cleanly
to the job description's language about "client context" and "business
insights."

All data is **100% synthetic**, generated by seed scripts. Never use real
business data.

## Stack вЂ” every choice is deliberate

### Ingestion (Bronze) вЂ” dlt
- **Why:** Python-native, handles schema evolution and incremental loading out
  of the box. Strong AI data engineering resume keyword. Free and local.
- **Swap path to prod:** Fivetran or Airbyte for managed connectors; keep dlt
  for custom API sources.
- **Do not** suggest replacing dlt with pandas scripts or SQLAlchemy. The point
  is the tool, not just the data movement.

### Table format вЂ” Delta Lake via delta-rs
- **Why:** Real lakehouse table format (ACID, time travel, partition pruning)
  without a JVM. `delta-rs` Python bindings run anywhere.
- **Swap path to prod:** Same Delta files read natively by Databricks/Spark.
  Zero conversion.
- **Do not** use Parquet files directly. Delta is required for the lakehouse
  story.

### Object storage вЂ” MinIO (S3-compatible)
- **Why:** Local S3 emulation so the code looks identical to production S3.
  Runs in Docker.
- **Swap path to prod:** Change one env var (`AWS_ENDPOINT_URL`) to point at
  real S3 or GCS.

### Operational source вЂ” Postgres
- **Why:** Represents a real operational database. Runs in Docker.
- **Swap path to prod:** Any production Postgres, MySQL, or cloud DB.

### Analytical engine вЂ” DuckDB
- **Why:** Reads Delta tables natively. Embedded, zero config, columnar
  vectorized execution. Free stand-in for Snowflake/BigQuery.
- **Swap path to prod:** Change `profiles.yml` to `dbt-snowflake` or
  `dbt-bigquery`. Model SQL is identical.
- **Do not** use SQLite. DuckDB's columnar engine and Delta support are the
  point.

### Transforms вЂ” dbt (dbt-duckdb)
- **Why:** Explicitly required in target job descriptions. Use dbt-duckdb
  adapter locally.
- **Required features to use:**
  - Model **contracts** on all Gold marts (enforce schema, document
    expectations)
  - Source **freshness** checks (define `loaded_at_field` and `warn`/`error`
    thresholds вЂ” these are the SLA mechanism)
  - **Generic tests** (`not_null`, `unique`, `accepted_values`,
    `relationships`) on all models
  - **Documentation** (`description:` on every model and column)
  - `dbt docs generate` output linked from README
- **Staging models** (Silver): one-to-one with sources, type casting,
  deduplication, null handling only
- **Mart models** (Gold): business logic, joins, aggregations, final grain

### Orchestration вЂ” Dagster
- **Why:** Software-defined assets map 1:1 onto medallion tiers (Bronze asset в†’
  Silver asset в†’ Gold asset). Built-in freshness checks and asset checks
  surface SLA compliance visually without extra tooling. Better demo than
  Airflow for a portfolio.
- **Airflow note:** Airflow appears in many JDs. ADR 003 documents the
  deliberate choice and notes the Airflow DAG equivalent. An
  `examples/airflow_equivalent.py` stub is acceptable to add.
- **Required Dagster features to use:**
  - `@asset` for each medallion layer
  - `FreshnessPolicy` on Silver and Gold assets
  - `AssetChecks` for row-count and schema validation
  - `ScheduleDefinition` for daily pipeline runs
  - Dagster UI screenshot in README showing green assets

### Data quality вЂ” Great Expectations
- **Why:** Industry-standard DQ tool. Expectation suites on Silver tables.
  Checkpoint results feed CI.
- **Required:** CI job must fail if GX checkpoint fails. This is the "CI gate"
  for data quality.
- **Swap path to prod:** Soda Core (same concept, different syntax вЂ” note in
  docs).

### Serving вЂ” FastAPI
- **Why:** Provides the AI-agent-ready context endpoint that proves this is an
  *AI* data engineering portfolio, not just a data engineering portfolio.
- **Required endpoint:**
  ```
  GET /context/{client_id}
  ```
  Returns a JSON snapshot of a client's current state (lead count, spend,
  top keywords, appointment rate) suitable for injection into an AI agent's
  context window. Target p99 latency: <50ms against DuckDB Gold tables.
- **Required:** Document p50 and p99 latency in README with actual measured
  numbers (run `locust` or a simple `time` loop and paste results).
- OpenAPI docs auto-generated by FastAPI at `/docs`.

### CI/CD вЂ” GitHub Actions
- **Required workflow steps (in order):**
  1. `ruff` lint + `black` format check
  2. `dbt compile` (catches SQL syntax errors)
  3. `dbt test` (schema tests against DuckDB)
  4. Great Expectations checkpoint
  5. `pytest` integration tests
  6. Docker build (confirms Dockerfile is valid)
- **Badge:** CI status badge must appear at the top of README.
- Workflow file: `.github/workflows/ci.yml`

## Architecture Decision Records

Every ADR lives in `docs/adr/` as a markdown file. Each follows this template:

```markdown
# ADR NNN вЂ” [title]

## Status
Accepted

## Context
[One paragraph: what problem were we solving, what constraints existed]

## Decision
[What we chose and why вЂ” be specific]

## Tradeoffs
[What we gave up. Be honest. This is what makes the ADR credible.]

## Swap path to production
[Exactly what changes when this project moves to a real cloud environment]
```

**Required ADRs:**

**ADR 001 вЂ” dlt over Fivetran/Airbyte**
Context: needed EL that runs locally with no accounts. Decision: dlt вЂ”
Python-native, incremental loading, schema evolution built in. Tradeoff:
Fivetran/Airbyte have 300+ pre-built connectors; dlt requires connector code.
Swap: managed connectors for operational sources; dlt stays for custom APIs.

**ADR 002 вЂ” DuckDB over Snowflake/BigQuery**
Context: needed analytical engine that's free and reads Delta natively.
Decision: DuckDB вЂ” embedded, zero config, columnar. Tradeoff: single-node, no
concurrent write scaling. Swap: only `profiles.yml` changes; dbt SQL is
identical.

**ADR 003 вЂ” Dagster over Apache Airflow**
Context: needed orchestrator that makes SLAs and lineage visible without extra
tooling. Decision: Dagster SDAs. Tradeoff: Airflow has larger ecosystem and is
explicitly in many JDs вЂ” this ADR documents the deliberate choice. Note: an
`examples/airflow_equivalent.py` stub can be added for keyword coverage.

**ADR 004 вЂ” delta-rs over PySpark**
Context: needed Delta table format without a JVM for local development.
Decision: delta-rs Python bindings. Tradeoff: Spark handles distributed Delta
writes at scale; delta-rs is single-node. Swap: same Delta files read natively
by Databricks/Spark вЂ” zero conversion needed.

## Code quality standards

- **Python:** `ruff` for linting, `black` for formatting, `mypy` for type hints
  on all public functions. `pyproject.toml` configures all three.
- **SQL:** All dbt models use CTEs (not subqueries). Every CTE is named and
  has a one-line comment. Final `SELECT` is always the last statement.
- **dbt:** Every model has a `description:` in schema.yml. Every column on
  Gold models has a `description:` and at least one test.
- **Docstrings:** Every Python function has a docstring. One-liners are fine
  for simple functions.
- **Tests:** `tests/test_contract_breaking.py` must exist and must demonstrate
  a failing CI check when a Gold model contract is violated.
- **No hardcoded credentials:** All secrets via `.env` (gitignored). `.env.example`
  committed with placeholder values.

## Data model вЂ” key tables

### Bronze (raw, Delta Lake on MinIO)

| Table | Source | Grain | Notes |
|---|---|---|---|
| `raw_clients` | Postgres | one row per client | append-only |
| `raw_leads` | Postgres | one row per lead event | append-only |
| `raw_appointments` | Postgres | one row per appointment | append-only |
| `raw_seo_rankings` | REST API | one row per client+keyword+week | append-only |
| `raw_ad_spend` | CSV | one row per client+channel+day | append-only |

### Silver (typed + validated, dbt staging, DuckDB)

Prefix: `stg_`. One-to-one with Bronze tables. Adds: correct types, surrogate
keys (`{{ dbt_utils.generate_surrogate_key([...]) }}`), `_loaded_at` timestamp,
deduplication via `row_number()` window.

### Gold (business-ready marts, dbt, DuckDB)

| Model | Grain | Key metrics |
|---|---|---|
| `dim_clients` | one row per client | plan tier, industry, days active |
| `fct_leads` | one row per lead | source, status, value, conversion flag |
| `fct_ad_performance` | client + channel + day | spend, impressions, clicks, CPL |
| `fct_appointments` | one row per appointment | lead-to-appt rate, channel |

## Synthetic data spec

Seed scripts must generate enough data to make queries interesting:

- **500 clients** across 5 industries (legal, dental, home services, med spa,
  auto) and 3 plan tiers (starter, pro, enterprise)
- **~50 leads per client** over 12 months вЂ” randomized sources (organic,
  paid, referral, social), statuses (new, contacted, qualified, converted,
  lost)
- **~5 appointments per client** per month (converted leads only)
- **52 SEO snapshots per client** (weekly) for 10 keywords each вЂ” position
  1вЂ“100 with realistic week-over-week drift
- **365 ad spend rows per client** (daily) for 3 channels (Google, Meta,
  email) вЂ” spend $10вЂ“$500/day with weekday/weekend pattern

All generators use `faker` + `random.seed(42)` for reproducibility. The same
`make seed` always produces identical data.

## Questions an AI should ask before writing any code

If any of the following is unclear, ask before proceeding:

1. Which specific file or component are we building right now?
2. Is this a new file or a modification to an existing one?
3. Should I generate realistic-looking synthetic data or minimal stub data?
4. Are we building the local version, the prod swap version, or both?
5. Should the output include the full file or just the changed section?

Do not make assumptions about scope. A request to "write the ingestion layer"
could mean one pipeline file or the entire `ingestion/` directory. Confirm.
